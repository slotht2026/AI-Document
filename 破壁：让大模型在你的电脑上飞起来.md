# 第一章：为什么你的大模型跑得慢？

> "我下载了一个 7B 的模型，结果每秒只出 2 个字，这正常吗？"
> ——每一个刚入门的人都说过这句话

---

## 1.1 一个简单的实验

让我们先做一个实验。假设你有以下配置：

| 硬件 | 规格 |
|------|------|
| CPU | Intel i7-12700K |
| 内存 | 32GB DDR4-3200 |
| 显卡 | NVIDIA RTX 3060 (12GB VRAM) |
| 硬盘 | 1TB NVMe SSD |

你下载了 Llama-2-7B 的 FP16 权重（约 14GB），用最朴素的方式加载运行。

**预期结果：** 每秒 1-3 个 token（约每秒 1-2 个汉字）

**体感：** 你在和一个反应迟钝的人聊天，每句话都要等十几秒。

这正常吗？**非常正常。** 但这是可以优化的，而且优化空间巨大——通常可以提升 5-20 倍。

---

## 1.2 推理慢的四个根本原因

### 原因一：模型太大，装不下

大模型的参数量直接决定了内存/显存需求：

| 模型 | 参数量 | FP16 大小 | FP32 大小 |
|------|--------|-----------|-----------|
| Qwen2.5-7B | 7B | 14GB | 28GB |
| Llama-3-8B | 8B | 16GB | 32GB |
| Qwen2.5-72B | 72B | 144GB | 288GB |
| DeepSeek-V3 | 671B | 1.3TB | 2.6TB |

你的 RTX 3060 只有 12GB 显存，连 7B 的 FP16 模型都装不下。系统被迫把部分权重放在 CPU 内存里，通过 PCIe 总线来回搬运——这就是第一个瓶颈。

**关键概念：内存墙（Memory Wall）**

大模型推理是典型的**内存带宽受限**（memory-bandwidth bound）任务。生成每个 token 时，GPU 需要从显存中读取几乎所有模型权重。RTX 3060 的显存带宽是 360 GB/s，听起来很快，但加载 14GB 的权重需要约 39ms，意味着理论上每秒最多生成约 25 个 token。而实际因为各种开销，通常只有理论值的 40-60%。

### 原因二：自回归生成，一次只出一个字

大语言模型使用**自回归解码**（autoregressive decoding）：每次只生成一个 token，然后把这个 token 加入输入，再生成下一个。

```
输入: "今天天气"
Step 1: 模型生成 "真"
Step 2: 输入变为 "今天天气真"，模型生成 "不"
Step 3: 输入变为 "今天天气真不"，模型生成 "错"
...
```

每一步都是一个完整的前向传播。这意味着：
- 无法像图像生成那样"并行出图"
- 每一步都有固定的延迟下限
- 序列越长，每一步的计算量越大（因为要处理所有历史 token）

### 原因三：注意力机制的 O(n²) 复杂度

Transformer 的注意力机制需要计算每个 token 与所有其他 token 的关系：

```
Attention(Q, K, V) = softmax(QK^T / √d_k) V
```

当序列长度为 n 时：
- 计算量：O(n²)
- 内存占用：O(n²)（存储注意力矩阵）
- KV Cache 大小：O(n)

一个 4096 token 的上下文，注意力矩阵有 16M 个元素。如果上下文扩展到 128K token，这个数字变成 160 亿。

### 原因四：硬件利用率低

这是最隐蔽的原因。你的 GPU 可能有很强的算力，但实际利用率很低：

- **CPU-GPU 数据搬运**：权重在 CPU 内存中，每次推理都要通过 PCIe 传输
- **小批量低效**：单条推理时，GPU 的数千个 CUDA 核心大部分在空转
- **内存碎片**：不合理的内存管理导致显存浪费
- **内核启动开销**：每个 CUDA 内核的启动都有微秒级开销，累积起来很可观

## 1.3 性能指标：怎么衡量"快不快"

在讨论加速之前，先明确我们优化的是什么：

### 指标一：吞吐量（Throughput）

单位时间处理的 token 数量，通常用 **tokens/s** 表示。

```
吞吐量 = 总生成 token 数 / 总耗时
```

### 指标二：首 Token 延迟（Time to First Token, TTFT）

从发送请求到收到第一个 token 的时间。这决定了用户感知的"响应速度"。

```
TTFT = Prefill 阶段耗时
```

### 指标三：每 Token 延迟（Inter-Token Latency, ITL）

生成相邻两个 token 之间的时间间隔。

```
ITL = 1 / 生成速度(tokens/s)
```

### 指标四：端到端延迟（End-to-End Latency）

从发送请求到收到完整回复的总时间。

```
端到端延迟 = TTFT + (生成 token 数 × ITL)
```

**不同场景关注不同指标：**
- 聊天机器人：TTFT 和 ITL 最重要（用户体验）
- 批量处理：吞吐量最重要（成本效率）
- 实时翻译：ITL 最重要（流畅度）

# 第二章：理解推理——从 Token 到响应的完整链路

> 不理解推理过程就去优化，就像不看地图就出发——你可能在反方向上跑得很快。

---

## 2.1 推理的三个阶段

一个完整的推理过程分为三个阶段：

```
用户输入 "请解释量子计算"
        │
        ▼
┌───────────────────┐
│  1. Tokenization  │  文本 → Token IDs
│     分词阶段       │  "请解释量子计算" → [8765, 23456, 1234, ...]
└───────┬───────────┘
        ▼
┌───────────────────┐
│  2. Prefill       │  Token IDs → 处理所有输入
│     预填充阶段     │  生成 KV Cache
│     (计算密集型)   │  输出第一个 token
└───────┬───────────┘
        ▼
┌───────────────────┐
│  3. Decode        │  逐个生成输出 token
│     解码阶段       │  每步用 KV Cache + 上一个 token
│     (内存带宽受限) │  直到生成 EOS 或达到长度限制
└───────┬───────────┘
        ▼
输出 "量子计算是利用量子力学原理..."
```

---

## 2.2 Tokenization：文本到数字

Tokenization 看似简单，但对推理性能有直接影响。

### 主流分词算法

| 算法 | 使用模型 | 特点 |
|------|----------|------|
| BPE (Byte Pair Encoding) | GPT 系列、Llama | 从字符开始合并高频对 |
| WordPiece | BERT | 类似 BPE，但用似然而非频率 |
| SentencePiece | Llama、Qwen | 语言无关，支持无空格语言 |
| Tiktoken | OpenAI 模型 | BPE 的高效实现 |

### Token 数量对性能的影响

Token 数量直接影响：
- **Prefill 耗时**：与输入 token 数成正比
- **KV Cache 大小**：与总 token 数成正比
- **注意力计算量**：与 token 数的平方成正比

**实用建议：**
- 中文通常比英文消耗更多 token（同一个意思）
- 选择对中文友好的分词器（如 Qwen 的 tiktoken-based 分词器）可以减少 30-50% 的 token 数
- 系统提示词（System Prompt）要精简，每个 token 都有成本

---

## 2.3 Prefill 阶段：并行处理所有输入

Prefill 是推理的第一个计算阶段，特点是：

- **可以高度并行**：所有输入 token 同时处理
- **计算密集型**（compute-bound）：主要瓶颈是算力，不是内存
- **一次性生成 KV Cache**：为后续 Decode 阶段准备

### Prefill 的计算过程

```
输入 Token IDs: [t₁, t₂, t₃, ..., tₙ]
         │
         ▼
   Embedding Layer
   token_id → 向量 (d_model 维)
         │
         ▼
   ┌─────────────────────────────┐
   │  Transformer Layer × L 层   │
   │                             │
   │  对于每一层:                 │
   │  1. 计算 Q, K, V 矩阵       │
   │  2. 计算注意力               │
   │  3. 存储 K, V 到 Cache      │
   │  4. FFN 前馈网络             │
   └─────────────────────────────┘
         │
         ▼
   LM Head (输出层)
         │
         ▼
   第一个输出 token
```

### Prefill 的计算量估算

对于一个 L 层、d_model 维、n 个输入 token 的模型：

```
Prefill FLOPs ≈ 2 × L × n × d_model² × (1 + 其他系数)
```

以 Llama-2-7B 为例（L=32, d_model=4096）：
- 处理 128 个 token：约 2.1 TFLOPs
- 处理 2048 个 token：约 33.5 TFLOPs
- 处理 32768 个 token：约 536 TFLOPs

**关键洞察：** Prefill 耗时与输入长度线性增长。长 Prompt 的 Prefill 可能需要几秒。

---

## 2.4 Decode 阶段：逐 Token 生成

Decode 是推理最核心、也是最慢的阶段。

### Decode 的计算过程

每一步：
1. 输入上一步生成的 token（单个 token）
2. 计算该 token 的 Q 向量
3. 用 Q 与 KV Cache 中的所有 K 计算注意力
4. 用注意力权重加权 V，得到输出
5. 通过 FFN 得到最终向量
6. LM Head 输出下一个 token 的概率分布
7. 采样得到下一个 token

```
第 i 步:
输入: token[i-1]
         │
         ▼
   Embedding → 向量
         │
         ▼
   ┌─────────────────────────────────┐
   │  每一层:                         │
   │  1. 计算 q_i, k_i, v_i          │
   │  2. 将 k_i, v_i 追加到 Cache    │
   │  3. attention = softmax(q_i · K^T) · V │
   │     (K, V 是整个 Cache)          │
   │  4. FFN                         │
   └─────────────────────────────────┘
         │
         ▼
   输出: token[i]
```

### 为什么 Decode 这么慢

**关键区别：** Prefill 是矩阵×矩阵（GEMM），Decode 是矩阵×向量（GEMV）。

```
Prefill: [n, d] × [d, d] = [n, d]  ← 大矩阵乘法，GPU 擅长
Decode:  [1, d] × [d, d] = [1, d]  ← 向量×矩阵，GPU 利用率低
```

GPU 有数千个核心，但单条 Decode 每步只处理一个 token 的向量。大部分核心在空转。

**这就是为什么 Decode 阶段是内存带宽受限的：**
- 需要从显存读取整个模型权重（几十 GB）
- 但实际计算量很小（一个向量的前向传播）
- GPU 算力严重浪费

### Decode 的时间分布

一个典型的 Decode 步骤的时间分布：

```
┌──────────────────────────────────────────────┐
│  权重加载 (从显存读取)     ████████████  75%  │
│  注意力计算                ██            10%  │
│  FFN 计算                  ███           12%  │
│  其他开销 (采样、通信等)    █              3%  │
└──────────────────────────────────────────────┘
```

**75% 的时间花在从显存读权重上！** 这就是为什么量化（减少权重大小）是最有效的加速手段。

---

## 2.5 KV Cache：空间换时间的典范

### 为什么需要 KV Cache

在 Decode 阶段，每生成一个新 token，都需要计算它与所有历史 token 的注意力。如果每次都重新计算所有历史 token 的 K 和 V，计算量会随序列长度急剧增长。

KV Cache 的解决方案：**缓存已经计算过的 K 和 V，每步只计算新 token 的 K 和 V。**

```
没有 KV Cache:
  Step 1: 计算 token_0 的 K, V                    → O(1)
  Step 2: 重新计算 token_0,1 的 K, V              → O(2)
  Step 3: 重新计算 token_0,1,2 的 K, V            → O(3)
  ...
  Step n: 重新计算 token_0..n-1 的 K, V           → O(n)
  总计: O(n²)

有 KV Cache:
  Step 1: 计算 token_0 的 K, V, 存入 Cache        → O(1)
  Step 2: 只计算 token_1 的 K, V, 追加到 Cache    → O(1)
  Step 3: 只计算 token_2 的 K, V, 追加到 Cache    → O(1)
  ...
  Step n: 只计算 token_n-1 的 K, V, 追加到 Cache  → O(1)
  总计: O(n)
```

### KV Cache 的内存占用

每个 Transformer 层的 KV Cache 大小：

```
KV Cache (单层) = 2 × n_tokens × d_head × n_heads × dtype_size
```

对于 Llama-2-7B（32 层，32 头，d_head=128，FP16）：

| 上下文长度 | KV Cache 大小 |
|-----------|---------------|
| 512 | 256 MB |
| 2048 | 1 GB |
| 8192 | 4 GB |
| 32768 | 16 GB |
| 131072 | 64 GB |

**注意：** 131K 上下文的 KV Cache 就要 64GB，比模型权重本身还大！

这就是为什么长上下文是推理加速的重要挑战，也是第五章的主题。

---

## 2.6 采样策略：最后一步也很关键

生成 token 概率分布后，需要从中采样。不同的采样策略影响输出质量，也影响计算开销。

### 常见采样方法

**贪心采样（Greedy）：**
```
token = argmax(probabilities)
```
最快，但输出单调重复。

**Top-K 采样：**
```
只保留概率最高的 K 个 token，重新归一化后采样
```
简单有效，但 K 是固定的。

**Top-P（Nucleus）采样：**
```
按概率从高到低累加，直到累积概率超过 P，只在这些 token 中采样
```
自适应候选集大小，更灵活。

**Temperature：**
```
probabilities = softmax(logits / temperature)
```
- temperature < 1：更确定，更保守
- temperature > 1：更多样，更随机
- temperature → 0：等价于贪心

**实际建议：** 对话场景用 temperature=0.7 + top_p=0.9；代码生成用 temperature=0 + 贪心。

## 2.7 端到端性能分析

让我们把所有阶段串起来，分析一个真实场景：

**场景：** 用 Llama-2-7B (FP16) 在 RTX 3060 (12GB) 上生成 512 个 token

```
Tokenization:     ~1ms        (可忽略)
Prefill (128 tok): ~200ms     (GPU 计算)
Decode (512 tok):  ~170s      (内存带宽受限)
  - 权重加载:      ~127s      (75%)
  - 注意力计算:    ~17s       (10%)
  - FFN 计算:      ~20s       (12%)
  - 其他:          ~5s        (3%)
采样:              ~2s        (可忽略)
─────────────────────────────
总计:              ~170s
速度:              ~3 tokens/s
```

**瓶颈一目了然：Decode 阶段的权重加载占了总时间的 75%。**

这就是为什么：
1. **量化**（减少权重大小）是最有效的加速手段——直接减少 75% 中的大部分
2. **批处理**（一次处理多个请求）可以分摊权重加载的成本
3. **投机解码**（用小模型预测大模型）可以减少大模型的 Decode 步数

---

## 2.8 Prefill vs Decode：两个不同的世界

| 特性 | Prefill | Decode |
|------|---------|--------|
| 输入 | n 个 token（整个 Prompt） | 1 个 token |
| 计算类型 | GEMM（矩阵乘法） | GEMV（矩阵向量乘法） |
| 瓶颈 | 计算密集 | 内存带宽密集 |
| 并行度 | 高（所有 token 并行） | 低（单 token） |
| 优化方向 | 算子融合、FlashAttention | 量化、减少权重大小 |
| 典型耗时占比 | 5-20% | 80-95% |

**实战启示：** 如果你的场景是短 Prompt + 长输出（如摘要生成），优化 Decode 是关键。如果是长 Prompt + 短输出（如文档问答），优化 Prefill 也很重要。

---

## 2.9 推理的内存全景

一次推理过程中，内存/显存的使用分布：

```
┌────────────────────────────────────────┐
│            GPU 显存使用                 │
├────────────────────────────────────────┤
│  模型权重          ████████████████     │ ← 最大头
│  KV Cache          ██████               │ ← 随上下文增长
│  激活值 (临时)      ██                   │ ← Prefill 时较大
│  框架开销           █                    │ ← 固定
│  空闲空间                              │ ← 越多越好
└────────────────────────────────────────┘
```

**显存不足时会发生什么？**

1. **模型权重被换出到 CPU 内存**：推理速度断崖式下降（PCIe 带宽 vs 显存带宽差 10-50 倍）
2. **KV Cache 被截断**：上下文长度受限
3. **OOM（Out of Memory）**：直接崩溃

# 第三章：量化——用精度换速度的艺术

> 量化的本质：你不需要用 64 位浮点数来存储一个只需要 4 位就能表达的数字。

---

## 3.1 什么是量化

量化（Quantization）是将模型权重从高精度（FP32/FP16）映射到低精度（INT8/INT4/NF4）的技术。

```
原始 (FP16):  0.0234375  →  16 bit
量化后 (INT8): 23         →  8 bit  (精度损失极小)
量化后 (INT4): 3          →  4 bit  (精度损失可控)
```

### 为什么量化有效

1. **减少内存占用**：模型体积缩小 2-8 倍
2. **减少内存带宽需求**：读取权重更快
3. **低精度计算更快**：INT8 矩阵乘法比 FP16 快 2-4 倍
4. **更多权重放入缓存**：减少缓存未命中

### 量化的基本数学

**线性量化公式：**

```
量化值 = round(原始值 / scale + zero_point)
反量化 = (量化值 - zero_point) × scale
```

其中：
- `scale = (max - min) / (2^n - 1)`，n 是量化位数
- `zero_point` 是零点偏移

**对称量化（更常用）：**

```
scale = max(|x|) / (2^(n-1) - 1)
量化值 = round(原始值 / scale)
```

---

## 3.2 量化类型详解

### 权重量化（Weight Quantization）

对模型权重进行量化，这是最常见、效果最直接的方式。

```
原始权重 (FP16):
┌─────────────────────────────────┐
│  0.023  -0.156  0.891  -0.034  │  每个值 16 bit
│  0.445  -0.678  0.123   0.567  │
└─────────────────────────────────┘

量化后 (INT8):
┌─────────────────────────────────┐
│   3     -20    114     -4      │  每个值 8 bit
│  57     -87     16     72      │
└─────────────────────────────────┘
```

### 权重+激活量化（Weight-Activation Quantization）

同时量化权重和激活值。激活值是中间计算结果，量化难度更大（因为有离群值）。

### KV Cache 量化

将 KV Cache 从 FP16 量化到 INT8 或 FP8，可以在相同显存下支持更长的上下文。

---

## 3.3 主流量化方法

### GPTQ（GPT 量化）

**原理：** 基于 OBQ（Optimal Brain Quantization），逐层量化，每次量化一个权重后调整其他权重来补偿误差。

**特点：**
- 需要校准数据集（几百条样本）
- 量化后模型质量高
- 支持 INT4/INT3
- 推理时需要反量化，有一定开销

**使用示例（AutoGPTQ）：**

```python
from transformers import AutoTokenizer
from auto_gptq import AutoGPTQForCausalLM

model_id = "TheBloke/Llama-2-7B-GPTQ"
tokenizer = AutoTokenizer.from_pretrained(model_id)
model = AutoGPTQForCausalLM.from_quantized(
    model_id,
    device="cuda:0",
    use_safetensors=True,
    use_triton=False,  # 不用 Triton 可以在更多平台运行
)
```

### AWQ（Activation-aware Weight Quantization）

**原理：** 只量化不重要的权重（对激活值影响小的），保留重要权重的精度。

**特点：**
- 比 GPTQ 更快（推理速度）
- 对离群激活值更友好
- 同样需要校准数据
- 是目前性价比最高的量化方法之一

**核心思想：**

```
重要权重 (占 1%)  →  保持高精度
普通权重 (占 99%) →  激进量化到 INT4
```

**使用示例：**

```python
from transformers import AutoTokenizer, AutoModelForCausalLM

model_id = "TheBloke/Llama-2-7B-AWQ"
tokenizer = AutoTokenizer.from_pretrained(model_id)
model = AutoModelForCausalLM.from_pretrained(
    model_id,
    device_map="auto",
    use_safetensors=True,
)
```

### GGUF（llama.cpp 的量化格式）

**原理：** 专门为 CPU/混合推理设计的量化格式，支持多种量化精度。

**特点：**
- 纯 CPU 推理友好
- 支持从 Q2_K 到 Q8_0 的多种精度
- 无需 GPU 即可运行
- 社区生态最活跃

**量化级别对比：**

| 量化类型 | 位数 | 大小(7B) | 质量 | 速度 |
|----------|------|----------|------|------|
| Q2_K | ~2.5 bit | 2.7 GB | 较差 | 最快 |
| Q3_K_S | ~3 bit | 3.2 GB | 一般 | 很快 |
| Q4_0 | 4 bit | 4.0 GB | 良好 | 快 |
| Q4_K_M | ~4.5 bit | 4.5 GB | 很好 | 快 |
| Q5_K_M | ~5.5 bit | 5.3 GB | 优秀 | 中等 |
| Q6_K | 6 bit | 5.9 GB | 极好 | 中等 |
| Q8_0 | 8 bit | 7.2 GB | 几乎无损 | 较慢 |

**使用 llama.cpp 量化：**

```bash
# 编译 llama.cpp
git clone https://github.com/ggerganov/llama.cpp
cd llama.cpp && make

# 量化模型
./llama-quantize input_model.gguf output_model.gguf Q4_K_M

# 运行推理
./llama-cli -m output_model.gguf -p "Hello" -n 100
```

### bitsandbytes（Hugging Face 生态）

**原理：** 使用 NF4（NormalFloat 4-bit）数据类型，基于正态分布优化的 4-bit 量化。

**特点：**
- 与 Hugging Face Transformers 深度集成
- 支持 QLoRA（量化 + 微调）
- 8-bit 量化几乎无损
- 4-bit 量化质量不错

**使用示例：**

```python
from transformers import AutoTokenizer, AutoModelForCausalLM, BitsAndBytesConfig

quantization_config = BitsAndBytesConfig(
    load_in_4bit=True,
    bnb_4bit_quant_type="nf4",       # NF4 比 FP4 更好
    bnb_4bit_compute_dtype="bfloat16", # 计算时用 bf16
    bnb_4bit_use_double_quant=True,   # 二次量化，进一步压缩
)

model = AutoModelForCausalLM.from_pretrained(
    "meta-llama/Llama-2-7b-hf",
    quantization_config=quantization_config,
    device_map="auto",
)
```

## 3.4 量化效果实测

以 Llama-2-7B 在 RTX 3060 (12GB) 上的实测数据：

| 精度 | 模型大小 | 显存占用 | 生成速度 | 质量评分 |
|------|----------|----------|----------|----------|
| FP16 | 14 GB | 15 GB | 无法加载 (OOM) | - |
| INT8 (bitsandbytes) | 7 GB | 8.5 GB | 8 tokens/s | 98/100 |
| GPTQ-4bit | 3.8 GB | 5.2 GB | 18 tokens/s | 95/100 |
| AWQ-4bit | 3.8 GB | 5.0 GB | 22 tokens/s | 96/100 |
| GGUF Q4_K_M | 4.5 GB | 5.5 GB | 15 tokens/s (GPU) | 95/100 |
| GGUF Q4_K_M | 4.5 GB | 4.5 GB | 6 tokens/s (CPU) | 95/100 |

**结论：** 4-bit 量化可以把速度提升 3-5 倍，质量损失几乎可以忽略。

## 3.5 量化的精度损失

量化不是免费午餐，会有精度损失。但好消息是：

### 损失分布不均匀

- **常识推理：** 几乎无损（INT4 和 FP16 差距 < 1%）
- **数学推理：** 有轻微损失（INT4 下降 2-5%）
- **代码生成：** 几乎无损
- **长文本生成：** 可能出现退化（累积误差）
- **低资源语言：** 损失较大

### 如何评估量化质量

```python
# 使用 lm-evaluation-harness 评估
# pip install lm-eval

import lm_eval

results = lm_eval.simple_evaluate(
    model="hf",
    model_args="pretrained=TheBloke/Llama-2-7B-GPTQ",
    tasks=["hellaswag", "arc_challenge", "mmlu"],
    batch_size=1,
)
```

## 3.6 混合精度量化

不是所有层都需要同样的精度。混合精度量化给不同层分配不同的位数：

```
Attention 层:  6 bit (对精度敏感)
FFN 前半部分:  4 bit
FFN 后半部分:  4 bit
Embedding:     8 bit (输入层，精度重要)
LM Head:       8 bit (输出层，精度重要)
```

**GGUF 的 K-quant 系统就是混合精度的典型实现：**

- Q4_K_M：重要层用 5-6 bit，普通层用 3-4 bit
- Q5_K_M：重要层用 6 bit，普通层用 4-5 bit

---

## 3.7 量化最佳实践

### 选择量化精度的决策树

```
显存够用？
├── 是 → 用 FP16/BF16（不量化）
└── 否 → 需要量化
         │
         ├── 显存差一点？→ INT8 量化
         │
         └── 显存差很多？→ INT4 量化
              │
              ├── 有 GPU？→ AWQ 或 GPTQ
              │
              └── 纯 CPU？→ GGUF Q4_K_M
```

### 量化工具选择

| 场景 | 推荐工具 | 原因 |
|------|----------|------|
| GPU 推理（NVIDIA） | AWQ | 速度最快，质量好 |
| GPU 推理（其他） | GPTQ | 兼容性更广 |
| CPU 推理 | GGUF (llama.cpp) | CPU 优化最好 |
| 混合 CPU+GPU | GGUF | 支持部分层卸载 |
| Hugging Face 生态 | bitsandbytes | 集成最方便 |
| 微调后量化 | GGUF 或 AWQ | 工具链成熟 |

### 常见陷阱

1. **不要对已经量化的模型再次量化**：质量会严重退化
2. **校准数据要代表性**：用代码数据校准的模型在对话上可能很差
3. **注意 tokenizer 兼容性**：不同量化工具可能修改 tokenizer
4. **检查特殊 token 处理**：量化后 EOS/BOS token 的行为可能变化

---

## 3.8 量化之外：剪枝与稀疏化

### 结构化剪枝

移除整个注意力头或 FFN 神经元：

```
原始模型: 32 层 × 32 头 = 1024 个注意力头
剪枝后:  32 层 × 24 头 = 768 个注意力头 (减少 25%)
```

### 非结构化稀疏化

将小权重设为零，利用稀疏矩阵计算加速：

```
原始: [0.02, 0.89, 0.001, -0.45, 0.003]
稀疏: [0,    0.89, 0,     -0.45, 0    ]
稀疏率: 60%
```

目前非结构化稀疏化在消费级 GPU 上的实际加速效果有限，因为主流 GPU 的稀疏计算支持还不够成熟。

## 4.2 llama.cpp——瑞士军刀

### 概述

llama.cpp 是 Georgi Gerganov 开发的纯 C/C++ 推理框架，是本地部署大模型的事实标准。

### 核心优势

1. **零依赖**：不需要 Python、CUDA、任何框架
2. **全平台**：Windows、Linux、macOS、Android、甚至浏览器 (WASM)
3. **全硬件**：CPU (AVX2/AVX-512)、NVIDIA GPU (CUDA)、AMD GPU (ROCm)、Apple GPU (Metal)、Intel GPU (SYCL)
4. **GGUF 格式**：最灵活的量化格式
5. **活跃社区**：每天都有 PR 合入

### 安装与使用

```bash
# 克隆并编译
git clone https://github.com/ggerganov/llama.cpp
cd llama.cpp

# CPU 编译 (最简单)
make -j

# CUDA 编译 (NVIDIA GPU)
GGML_CUDA=1 make -j

# Metal 编译 (Apple Silicon)
GGML_METAL=1 make -j

# 运行推理
./llama-cli -m model.gguf \
    -p "Tell me a joke" \
    -n 200 \
    --temp 0.7

# 启动 API 服务 (兼容 OpenAI API)
./llama-server -m model.gguf --port 8080
```

### GPU Offload

llama.cpp 支持将部分层放到 GPU，其余留在 CPU：

```bash
# 将所有层放到 GPU
./llama-cli -m model.gguf -ngl 99

# 只放 20 层到 GPU (显存不够时)
./llama-cli -m model.gguf -ngl 20

# 纯 CPU
./llama-cli -m model.gguf -ngl 0
```

**实用技巧：** 先用 `nvidia-smi` 看可用显存，然后计算能放多少层。7B 模型 32 层，Q4_K_M 量化后每层约 120MB。12GB 显存可以放约 30 层。

### 性能调优参数

```bash
./llama-cli -m model.gguf \
    -ngl 99 \              # GPU 层数
    -t 8 \                 # CPU 线程数 (通常设为物理核心数)
    -b 512 \               # 批处理大小
    -ub 512 \              # prompt 批处理大小
    --mlock \              # 锁定内存，防止换页
    --no-mmap \            # 不使用内存映射 (有时更快)
    -ctk q8_0 \            # KV Cache 量化 (Key)
    -ctv q8_0              # KV Cache 量化 (Value)
```

## 4.4 vLLM——服务器级推理引擎

### 概述

vLLM 是 UC Berkeley 开发的高性能推理引擎，专为服务器场景设计。

### 核心技术：PagedAttention

传统 KV Cache 需要连续内存，导致大量浪费。PagedAttention 借鉴操作系统的虚拟内存思想：

```
传统 KV Cache:
┌─────────────────────────────────────┐
│ [请求1 KV Cache    ][空闲][空闲]    │  ← 连续分配，有碎片
│ [请求2 KV Cache         ][空闲]     │
└─────────────────────────────────────┘

PagedAttention:
┌─────────────────────────────────────┐
│ [页1][页3][页5][页7][页2][页4][页6] │  ← 分页管理，无碎片
└─────────────────────────────────────┘
每个请求的 KV Cache 可以存在不连续的物理页中
通过页表映射到连续的逻辑地址
```

### 核心优势

1. **连续批处理**（Continuous Batching）：请求可以随时加入/离开批次
2. **PagedAttention**：显存利用率提升 2-4 倍
3. **张量并行**：多 GPU 推理
4. **高吞吐**：比 Hugging Face Transformers 快 14-24 倍

### 使用示例

```python
# 安装
# pip install vllm

from vllm import LLM, SamplingParams

# 初始化
llm = LLM(
    model="Qwen/Qwen2.5-7B-Instruct",
    tensor_parallel_size=1,       # GPU 数量
    gpu_memory_utilization=0.9,   # 显存利用率
    max_model_len=8192,           # 最大上下文长度
)

# 采样参数
sampling_params = SamplingParams(
    temperature=0.7,
    top_p=0.9,
    max_tokens=512,
)

# 批量推理
prompts = [
    "解释量子计算",
    "写一首关于AI的诗",
    "Python 快速排序代码",
]

outputs = llm.generate(prompts, sampling_params)

for output in outputs:
    print(output.outputs[0].text)
```

### 启动 API 服务

```bash
python -m vllm.entrypoints.openai.api_server \
    --model Qwen/Qwen2.5-7B-Instruct \

# 第五章：KV Cache——被忽视的性能杀手

> 模型权重决定了你能不能跑，KV Cache 决定了你能跑多长。

---

## 5.1 KV Cache 的内存困境

回顾第二章的计算：Llama-2-7B 在 128K 上下文下，KV Cache 需要 64GB——比模型权重本身还大。

这不是理论问题，而是实际部署中最常遇到的瓶颈：

```
RTX 3060 (12GB VRAM):
├── 模型权重 (Q4_K_M):  4.5 GB
├── KV Cache (2048 tok): 1.0 GB
├── 框架开销:            0.5 GB
└── 剩余:               6.0 GB → 最大约 12K 上下文

RTX 3060 (12GB VRAM):
├── 模型权重 (Q4_K_M):  4.5 GB
├── KV Cache (32K tok):  16 GB  ← 超了！
└── OOM!
```

---

## 5.2 KV Cache 内存计算公式

精确计算 KV Cache 的内存需求：

```
KV Cache (bytes) = 2 × n_layers × n_heads × d_head × seq_len × dtype_bytes
```

其中：
- 2 = K 和 V 各一份
- n_layers = Transformer 层数
- n_heads = 注意力头数（或 KV 头数，对 GQA 模型）
- d_head = 每个头的维度
- seq_len = 序列长度
- dtype_bytes = 数据类型字节数（FP16=2, FP8=1, INT8=1）

**GQA（Grouped Query Attention）模型的 KV Cache 更小：**

GQA 中 KV 头数 < Q 头数，KV Cache 按 KV 头数计算：

```
Llama-2-7B:  32 层 × 32 KV 头 × 128 维 = 每 token 256 KB
Llama-3-8B (GQA): 32 层 × 8 KV 头 × 128 维 = 每 token 64 KB  ← 小 4 倍
```

---

## 5.3 KV Cache 量化

最直接的优化：把 KV Cache 从 FP16 量化到更低精度。

### 量化方案对比

| 精度 | 每 token 大小 (7B) | 质量影响 |
|------|-------------------|----------|
| FP16 | 256 KB | 基准 |
| FP8 | 128 KB | 几乎无损 |
| INT8 | 128 KB | 几乎无损 |
| Q4_K | ~80 KB | 轻微损失 |
| Q2_K | ~48 KB | 可感知损失 |

### llama.cpp 中启用 KV Cache 量化

```bash
# FP8 KV Cache
./llama-cli -m model.gguf \
    -ctk q8_0 \   # Key cache 量化为 Q8_0
    -ctv q8_0     # Value cache 量化为 Q8_0

# 更激进的量化
./llama-cli -m model.gguf \
    -ctk q4_0 \   # Key cache 量化为 Q4_0
    -ctv q4_0     # Value cache 量化为 Q4_0
```

**实测效果：** Q8_0 KV Cache 量化可以在几乎无质量损失的情况下，将 KV Cache 内存减半。

---

## 5.4 滑动窗口注意力

### 原理

不是所有历史 token 都同等重要。滑动窗口注意力只保留最近的 W 个 token 的 KV Cache：

```
完整注意力 (seq_len = 1000):
[token_0] [token_1] ... [token_998] [token_999]
←────────── 全部参与注意力 ──────────→

滑动窗口 (W = 512):
[token_0] ... [token_487] [token_488] ... [token_999]
←── 丢弃 ──→←────── 保留窗口 ──────→
```

### 实现方式

**Mistral 的滑动窗口：**

Mistral-7B 使用滑动窗口注意力，每层只关注最近的 4096 个 token。但通过多层堆叠，信息可以逐层传播到更远的位置：

```
Layer 1: 看到最近 4096 token
Layer 2: 通过 Layer 1 间接看到最近 8192 token
Layer k: 间接看到最近 k × 4096 token
```

### llama.cpp 中的上下文压缩

```bash
# 启用上下文压缩 (当上下文接近最大值时自动压缩)
./llama-cli -m model.gguf \
    --ctx-size 8192 \
    --cache-type-k q8_0 \
    --cache-type-v q8_0
```

## 5.5 KV Cache 复用

### Prefix Caching

多个请求共享相同的系统提示时，可以复用其 KV Cache：

```
请求 1: [系统提示] + [用户问题 A]
请求 2: [系统提示] + [用户问题 B]
                ↑
        这部分 KV Cache 可以共享
```

**vLLM 的 Automatic Prefix Caching：**

```python
from vllm import LLM

llm = LLM(
    model="Qwen/Qwen2.5-7B-Instruct",
    enable_prefix_caching=True,  # 启用前缀缓存
)
```

**SGLang 的 RadixAttention：**

SGLang 使用基数树管理 KV Cache，自动发现和复用公共前缀：

```python
import sglang as sgl

@sgl.function
def chat(s, question):
    s += sgl.system("You are a helpful assistant.")  # 共享部分
    s += sgl.user(question)
    s += sgl.assistant(sgl.gen("answer", max_tokens=512))

# 多个请求自动共享系统提示的 KV Cache
```

## 5.6 KV Cache 压缩技术

### H₂O（Heavy-Hitter Oracle）

**原理：** 只保留"重要"token 的 KV Cache。重要性由注意力分数决定。

```
完整 Cache: [t0, t1, t2, t3, t4, t5, t6, t7, t8, t9]
注意力分数: [0.01, 0.02, 0.15, 0.03, 0.25, 0.01, 0.30, 0.02, 0.18, 0.03]
                                                  ↑ 重要 token
压缩后:     [t2, t4, t6, t8]  (保留注意力分数最高的)
```

### StreamingLLM

**原理：** 发现前几个 token（"注意力汇聚点"）对生成质量极其重要，必须保留。结合滑动窗口：

```
[初始 token (必须保留)] + [最近的窗口]
     ↑                      ↑
  通常前 4 个           滑动窗口内的 token
```

## 5.7 多查询注意力 (MQA) 与分组查询注意力 (GQA)

### 从 MHA 到 GQA 的演进

```
MHA (Multi-Head Attention):
  Q heads: 32
  K heads: 32  ← KV Cache 大
  V heads: 32

MQA (Multi-Query Attention):
  Q heads: 32
  K heads: 1   ← KV Cache 极小，但质量下降
  V heads: 1

GQA (Grouped Query Attention):
  Q heads: 32
  K heads: 8   ← KV Cache 较小，质量几乎无损
  V heads: 8
```

### KV Cache 大小对比 (Llama 系列)

| 模型 | 架构 | KV 头数 | 每 token KV Cache |
|------|------|---------|-------------------|
| Llama-2-7B | MHA | 32 | 256 KB |
| Llama-3-8B | GQA | 8 | 64 KB |
| Llama-3-70B | GQA | 8 | 128 KB |

**GQA 是目前的主流选择：** KV Cache 减少 4 倍，质量几乎无损。

## 5.8 实战：KV Cache 容量规划

### 计算最大上下文长度

```python
def max_context_length(
    vram_gb: float,
    model_size_gb: float,
    n_layers: int,
    n_kv_heads: int,
    d_head: int,
    dtype_bytes: int = 2,  # FP16
    overhead_gb: float = 1.0,
) -> int:
    """计算给定显存下支持的最大上下文长度"""
    available_gb = vram_gb - model_size_gb - overhead_gb
    bytes_per_token = 2 * n_layers * n_kv_heads * d_head * dtype_bytes
    max_tokens = int(available_gb * 1e9 / bytes_per_token)
    return max_tokens

# Llama-2-7B on RTX 3060 (12GB)
max_ctx = max_context_length(
    vram_gb=12,
    model_size_gb=4.5,  # Q4_K_M
    n_layers=32,
    n_kv_heads=32,
    d_head=128,
    dtype_bytes=2,
)
print(f"最大上下文: {max_ctx} tokens")  # 约 18000

# 启用 KV Cache 量化 (FP8)
max_ctx_q8 = max_context_length(
    vram_gb=12,
    model_size_gb=4.5,
    n_layers=32,
    n_kv_heads=32,
    d_head=128,
    dtype_bytes=1,  # FP8
)
print(f"FP8 KV Cache 最大上下文: {max_ctx_q8} tokens")  # 约 36000
```

# 第六章：投机解码——让小模型替大模型干活

> 核心思想：用一个快但不太准的小模型"猜"多个 token，然后让大模型一次性验证。

---

## 6.1 基本原理

传统自回归解码的瓶颈在于：大模型每次只能生成一个 token。

投机解码（Speculative Decoding）的思路：

```
传统方式:
  大模型: token_1 → token_2 → token_3 → token_4 → token_5
  (每步都慢)

投机解码:
  小模型: 快速猜出 [token_1, token_2, token_3, token_4, token_5]
  大模型: 一次性验证 [✓, ✓, ✗, ✓, ✓]
  接受前 2 个，从 ✗ 处重新开始
  (大模型只跑了 1 次，得到 2+ 个 token)
```

### 数学保证

投机解码的关键特性：**输出分布与纯大模型解码完全一致。**

这不是近似，是精确等价。通过精心设计的接受-拒绝采样策略：

```
对于小模型预测的 token x:
  概率 p_small(x) ≥ p_large(x) → 一定接受
  概率 p_small(x) < p_large(x) → 以 p_small(x)/p_large(x) 的概率接受
  拒绝 → 从修正分布中重新采样
```

---

## 6.2 实现方式

### 方式一：独立小模型

使用一个同系列但更小的模型作为草稿模型：

```
大模型: Llama-2-70B (慢但准)
小模型: Llama-2-7B (快但不太准)

小模型快速生成 5 个候选 token
大模型一次性验证 (并行计算，和处理 5 个 token 的 Prompt 一样快)
```

**优点：** 实现简单，效果稳定
**缺点：** 需要额外加载小模型，增加内存占用

### 方式二：Medusa（多头预测）

在大模型上添加多个"预测头"，每个头预测未来不同位置的 token：

```
大模型最后一层隐藏状态 h
  ├── Medusa Head 1 → 预测 token+1
  ├── Medusa Head 2 → 预测 token+2
  ├── Medusa Head 3 → 预测 token+3
  └── Medusa Head 4 → 预测 token+4
```

**优点：** 不需要额外模型
**缺点：** 需要训练 Medusa 头，预测准确度较低

### 方式三：EAGLE（自回归草稿）

使用一个轻量级的自回归头，基于大模型的特征进行预测：

```
大模型特征 → EAGLE 头 → 自回归生成草稿序列
                        (比独立小模型更准)
```

**优点：** 准确度高，加速比大
**缺点：** 需要训练 EAGLE 头

## 6.3 llama.cpp 中的投机解码

```bash

    -md llama-2-7b-q4_k_m.gguf \        # 草稿模型

```

**参数调优：**

```bash

# - 值太小: 加速不明显
# - 值太大: 接受率低，浪费计算

# draft-p-min: 最小接受概率阈值
# - 更高的值: 更保守，接受率更高
# - 推荐: 0.1-0.3
```

## 6.4 加速效果

### 理论加速比

假设：
- 小模型速度是大模型的 k 倍
- 小模型预测的接受率为 α
- 每次猜测 n 个 token

```
有效加速比 ≈ (1 - α^(n+1)) / (1 - α) × (k / (k + n))
```

### 实测数据

| 大模型 | 草稿模型 | 硬件 | 加速比 |
|--------|----------|------|--------|
| Llama-2-70B | Llama-2-7B | RTX 4090 | 2.5x |
| Qwen-72B | Qwen-7B | A100 | 2.8x |
| Llama-3-70B | Llama-3-8B | RTX 4090 | 3.0x |
| Mixtral-8x7B | Llama-3-8B | RTX 4090 | 2.2x |

**关键观察：**
- 小模型与大模型越相似，接受率越高
- 代码生成的接受率通常高于对话
- 事实性问题的接受率低于创意写作

## 6.5 Self-Speculative Decoding

不需要额外模型或头，大模型自己做投机：

### 思路

跳过某些 Transformer 层，用浅层特征快速生成候选：

```
完整模型 (32 层): 慢但准
跳层模型 (16 层): 快但不太准 → 用于草稿
完整模型 (32 层): 验证
```

## 6.6 Lookahead Decoding

### 原理

不用小模型，而是利用 Jacobi 迭代的思想，将自回归生成转化为并行求解：

```
传统: x1 → x2 → x3 → x4 (串行)

Lookahead:
  初始猜测: [x1?, x2?, x3?, x4?]
  迭代 1:   [x1, x2?, x3?, x4?]  (x1 确定)
  迭代 2:   [x1, x2, x3?, x4?]   (x2 确定)
  迭代 3:   [x1, x2, x3, x4?]    (x3 确定)
```

**优点：** 不需要任何额外模型
**缺点：** 加速比通常不如投机解码（1.5x-2x）

---

## 6.7 选择哪种投机解码方法

| 方法 | 额外内存 | 训练需求 | 加速比 | 实现复杂度 |
|------|----------|----------|--------|------------|
| 独立小模型 | 高 (加载小模型) | 无 | 2-3x | 低 |
| Medusa | 低 (几个头) | 需要训练 | 2-3x | 中 |
| EAGLE | 低 (一个头) | 需要训练 | 3-4x | 中 |
| Self-Speculative | 无 | 无 | 1.5-2x | 中 |
| Lookahead | 无 | 无 | 1.5-2x | 低 |

**推荐策略：**
1. 如果显存充足 → 独立小模型（最简单）
2. 如果显存紧张 → Self-Speculative 或 Lookahead
3. 如果愿意训练 → EAGLE（效果最好）

---

## 6.8 投机解码的局限

1. **小模型质量影响加速比**：小模型越差，接受率越低，加速越小
2. **增加显存占用**：独立小模型需要额外显存
3. **批量推理效果减弱**：投机解码对单条请求效果最好，批处理时优势减少
4. **不适用于所有任务**：高度随机的任务（创意写作）接受率低

|------|----------|

| 显存有限 | Self-Speculative 或 Lookahead |
| 愿意训练 | EAGLE |
| llama.cpp 用户 | --draft 参数 |
| vLLM 用户 | 开启 speculative_decoding |

# 第七章：批处理与并发——榨干最后一滴算力

> 单条请求的 GPU 利用率可能只有 5%。批处理的本质是让更多请求分摊同一份权重加载的开销。

---

## 7.1 为什么批处理能加速

回顾第二章的结论：Decode 阶段 75% 的时间花在加载模型权重上。

**单条请求：**
```
加载 14GB 权重 → 计算 1 个 token → 输出
         ↑
    75% 的时间
```

**8 条请求批处理：**
```
加载 14GB 权重 → 计算 8 个 token → 输出 8 条
         ↑
    同样的时间，8 倍产出
```

权重加载的时间是固定的，不管处理 1 个还是 8 个 token，都要加载同样的权重。

---

## 7.2 Static Batching vs Continuous Batching

### Static Batching

传统的批处理方式：凑满一个批次才开始处理。

```
请求 1: [============]  (4s)
请求 2: [========]      (3s)    ← 等待请求 1 完成
请求 3: [==========]    (3.5s)  ← 等待请求 1 完成
                            ↑
                    所有请求同步结束
                    短请求要等长请求
```

### Continuous Batching

vLLM 引入的方式：请求可以随时加入/离开批次。

```
请求 1: [============]
请求 2:     [========]
请求 3:          [==========]
请求 4:              [====]
                             ↑
                    每个请求独立完成
                    没有等待浪费
```

**效果：** 吞吐量提升 2-4 倍。

---

## 7.3 vLLM 的批处理实践

### 启动高吞吐服务

```python

## 7.4 llama.cpp 的批处理

### 多 slot 支持

llama.cpp 的 server 模式支持多 slot（并发请求）：

```bash
./llama-server \
    -m model.gguf \
    -ngl 99 \
    -c 8192 \          # 上下文大小
    -np 8 \            # 8 个并发 slot
    -b 512 \           # 批处理大小
    --cont-batching    # 启用连续批处理
```

### Slot 与上下文的关系

每个 slot 需要独立的 KV Cache：

```
总上下文 = 每个 slot 的上下文 × slot 数
8192 × 8 = 65536 tokens 的 KV Cache
```

如果显存不够，需要减小每个 slot 的上下文或 slot 数。

## 7.5 Prefill 与 Decode 的分离

### Chunked Prefill

长 Prompt 的 Prefill 会阻塞 Decode。Chunked Prefill 将 Prefill 切成小块，与 Decode 交错执行：

```
传统:
  [====Prefill (长 Prompt)====][Decode][Decode]...
                                ↑ 被阻塞

Chunked Prefill:
  [Prefill chunk1][Decode][Prefill chunk2][Decode]...
                   ↑ 不被阻塞
```

**效果：** 降低 Decode 请求的延迟抖动。

### Disaggregated Prefill

将 Prefill 和 Decode 放在不同的 GPU 上：

```
GPU 1 (Prefill 专用): 处理所有新请求的 Prefill
GPU 2 (Decode 专用):  处理所有请求的 Decode
```

**适用场景：** 多 GPU 服务器，Prefill 和 Decode 的负载特性不同。

---

## 7.6 并发数的选择

### 吞吐量 vs 延迟的权衡

```
并发数:  1    4    8    16   32   64
         │    │    │    │    │    │
吞吐量:  低 ──────────────────→ 高
延迟:    低 ──────────────────→ 高
         │    │    │    │    │    │
GPU利用率: 5%  20%  40%  70%  90%  95%
```

**推荐策略：**
- 追求低延迟：并发数 1-4
- 追求高吞吐：并发数 16-32
- 平衡：并发数 8-16

### 显存与并发的关系

```
每个请求的 KV Cache: seq_len × 每 token 大小
总 KV Cache: 并发数 × 每请求 KV Cache

例: 8 并发 × 4096 上下文 × 64KB/token = 2GB KV Cache
```

---

## 7.7 动态批处理策略

### 自适应批处理大小

根据当前负载动态调整批处理大小：

```python
def adaptive_batch_size(
    pending_requests: int,
    avg_seq_len: int,
    max_batch_tokens: int,
    max_concurrent: int,
) -> int:
    """根据负载动态计算批处理大小"""
    # 基于 token 预算
    token_based = max_batch_tokens // avg_seq_len
    # 基于并发限制
    concurrent_based = min(pending_requests, max_concurrent)
    # 取较小值
    return min(token_based, concurrent_based)
```

### 优先级调度

给不同请求分配优先级：

```
高优先级: 实时聊天 (低延迟)
中优先级: API 调用 (适中延迟)
低优先级: 批量处理 (高吞吐)
```

## 7.8 多 GPU 批处理

### 张量并行 (Tensor Parallelism)

将模型权重切分到多个 GPU：

```
GPU 0: 模型的前半部分层
GPU 1: 模型的后半部分层

请求 → GPU 0 计算前半 → GPU 1 计算后半 → 输出
```

**vLLM 配置：**

```python
llm = LLM(
    model="Qwen/Qwen2.5-72B-Instruct",
    tensor_parallel_size=2,  # 2 个 GPU
)
```

### 流水线并行 (Pipeline Parallelism)

将模型的不同层分配到不同 GPU，请求在 GPU 间流水线传递。

**适用场景：** 模型太大，单 GPU 放不下。

# 第八章：模型架构优化——从源头瘦身

> 最快的计算是不需要计算。最省内存的权重是不需要权重。

---

## 8.1 为什么架构选择比推理优化更重要

假设你在两台电脑上分别运行：

- 电脑 A：Llama-2-7B (MHA, 32 层) + 所有推理优化
- 电脑 B：Qwen2.5-7B (GQA, 28 层) + 基础推理

电脑 B 可能比电脑 A 更快，因为：
1. GQA 的 KV Cache 更小
2. 更少的层意味着更少的计算
3. 更高效的注意力实现

**架构优化是乘法效应：** 好的架构 × 推理优化 = 最佳性能。

---

## 8.2 注意力机制的演进

### 标准 Multi-Head Attention (MHA)

```
每个注意力头独立计算 Q, K, V:
  Q = X × W_q  (32 个头)
  K = X × W_k  (32 个头)
  V = X × W_v  (32 个头)
```

**问题：** KV Cache 头数多，内存占用大。

### Multi-Query Attention (MQA)

所有 Q 头共享同一组 K, V：

```
  Q = X × W_q  (32 个头)
  K = X × W_k  (1 个头)  ← 共享
  V = X × W_v  (1 个头)  ← 共享
```

**效果：** KV Cache 减少 32 倍，但质量有下降。

### Grouped Query Attention (GQA)

折中方案：Q 头分组，每组共享 K, V：

```
  Q = X × W_q  (32 个头，分 8 组)
  K = X × W_k  (8 个头)
  V = X × W_v  (8 个头)
```

**效果：** KV Cache 减少 4 倍，质量几乎无损。这是目前的主流。

### Multi-head Latent Attention (MLA)

DeepSeek-V2 提出的方案，将 KV 压缩到低维空间：

```
  K = compress(X × W_k)  → 低维表示
  V = compress(X × W_v)  → 低维表示
  需要时再解压
```

**效果：** KV Cache 减少 10 倍以上，质量保持。

---

## 8.3 前馈网络 (FFN) 的优化

### 标准 FFN

```
FFN(x) = GELU(x × W1) × W2

W1: [d_model, d_ff]  (通常 d_ff = 4 × d_model)
W2: [d_ff, d_model]
```

### SwiGLU (Llama、Qwen 使用)

```
FFN(x) = (SiLU(x × W_gate) ⊙ (x × W_up)) × W_down

三个权重矩阵，但 d_ff 可以更小 (通常 8/3 × d_model)
```

**效果：** 参数量相近，但效果更好。

### Mixture of Experts (MoE)

不是所有参数都激活，只激活一部分"专家"：

```
输入 → Router → 选择 2 个专家 (从 8 个中)
                ├── Expert 1 计算
                └── Expert 3 计算
输出 = Expert1_output + Expert3_output
```

**关键指标：** 激活参数量 vs 总参数量

| 模型 | 总参数 | 激活参数 | 等效模型 |
|------|--------|----------|----------|
| Mixtral-8x7B | 47B | 13B | ~13B 密集模型 |
| DeepSeek-V3 | 671B | 37B | ~37B 密集模型 |
| Qwen2.5-MoE | 14B | 3B | ~3B 密集模型 |

**对推理的意义：** MoE 模型的推理速度取决于激活参数量，不是总参数量。但内存需要装下所有专家权重。

---

## 8.4 位置编码的优化

### RoPE (Rotary Position Embedding)

目前最主流的位置编码，但长上下文外推是个问题：

```
训练长度: 4096
推理时: 想用 32768

问题: RoPE 在训练长度之外的位置编码质量下降
```

### YaRN (Yet another RoPE extensioN)

通过温度缩放和频率修改，让 RoPE 支持更长的上下文：

```python
# YaRN 的核心：对不同频率分量应用不同的缩放
def yarn_rope(theta, d, position):
    # 低频分量: 线性插值
    # 高频分量: 不缩放
    # 中频分量: 渐进缩放
    ...
```

### NTK-aware Scaling

另一种 RoPE 扩展方法，修改 base frequency：

```python
# 将 RoPE 的 base 从 10000 扩大
new_base = original_base * (scale_factor ** (d / (d - 2)))
```

**llama.cpp 中的上下文扩展：**

```bash
# 使用 YaRN 扩展上下文
./llama-cli -m model.gguf \
    --ctx-size 32768 \
    --rope-scaling yarn \
    --rope-freq-scale 0.25
```

## 8.5 高效 Transformer 架构

### Linear Attention

将 O(n²) 注意力替换为 O(n) 线性注意力：

```
标准: Attention = softmax(QK^T) V  → O(n²)
线性: Attention = φ(Q) (φ(K)^T V)  → O(n) (通过改变计算顺序)
```

**问题：** 效果通常不如标准注意力。

### State Space Models (Mamba)

完全不同的序列建模方式，用状态空间替代注意力：

```
Transformer: 每个 token 看到所有历史 token (O(n²))
Mamba:       通过隐状态传递信息 (O(n))

Mamba 的优势:
  - 线性复杂度
  - 无限上下文长度 (理论上)
  - 推理时内存固定

Mamba 的劣势:
  - 长距离信息检索不如 Transformer
  - "大海捞针"测试表现较弱
```

### Jamba (Mamba + Transformer 混合)

AI21 的方案：交替使用 Mamba 层和 Transformer 层：

```
Layer 1: Mamba (高效序列建模)
Layer 2: Mamba
Layer 3: Transformer (精确注意力)
Layer 4: Mamba
...
```

**效果：** 结合两者优势，推理效率高且质量好。

## 8.6 模型压缩：知识蒸馏

用大模型训练小模型，让小模型学到大模型的能力：

```
教师模型 (70B)
    │
    │ 知识蒸馏
    ▼
学生模型 (7B)  → 性能接近 14B，推理速度接近 7B
```

**实际案例：**
- Gemma 2B/7B：从更大的 Gemini 模型蒸馏
- Phi-3 Mini (3.8B)：从 GPT-4 级别模型蒸馏
- Qwen2.5 系列：不同尺寸间有蒸馏关系

## 9.1 CPU 调优

### 线程数选择

llama.cpp 的 `-t` 参数控制 CPU 线程数：

```
最佳线程数 = 物理核心数（不是逻辑核心数）

i7-12700K: 12 物理核心 → -t 12
Ryzen 9 7950X: 16 物理核心 → -t 16
M2 Pro: 10 物理核心 → -t 10
```

**为什么不用超线程？** 大模型推理的计算密集度高，超线程的收益很小，反而可能因为资源争抢而变慢。

### NUMA 优化

多路 CPU 服务器要注意 NUMA（非统一内存访问）：

```bash
# 查看 NUMA 拓扑
numactl --hardware

# 绑定到单个 NUMA 节点
numactl --cpunodebind=0 --membind=0 ./llama-cli -m model.gguf
```

### CPU 缓存优化

```bash
# 使用 mlock 锁定内存，防止换页
./llama-cli -m model.gguf --mlock

# 禁用 mmap（有时可以减少缓存抖动）
./llama-cli -m model.gguf --no-mmap
```

### CPU 指令集

确保编译时启用了正确的指令集：

```bash
# 检查 CPU 支持的指令集
cat /proc/cpuinfo | grep flags

# llama.cpp 编译时启用 AVX2
make -j LLAMA_NATIVE=1

# 如果支持 AVX-512
make -j LLAMA_NATIVE=1 LLAMA_AVX512=1
```

## 12.2 模型架构趋势

### 更小但更强的模型

```
2023: 7B 模型 ≈ 2022 年的 13B 模型
2024: 3B 模型 ≈ 2023 年的 7B 模型
2025: 1B 模型接近 2023 年的 7B 模型

趋势: 同等能力的模型每年缩小约 2 倍
```

### 新架构探索

- **Mamba/State Space Models**：线性复杂度，长序列友好
- **RWKV**：RNN 架构但有 Transformer 级别的性能
- **Hyena**：长卷积替代注意力
- **RetNet**：保留机制替代注意力

### MoE 的普及

```
趋势: 更多模型使用 MoE 架构

优势:
  - 同等性能下，激活参数更少
  - 推理更快 (只激活部分专家)
  - 训练更高效

挑战:
  - 需要更多显存 (所有专家权重)
  - 路由策略复杂
  - 批处理效率可能降低
```

## 12.3 量化技术前沿

### 1-bit 模型 (BitNet)

微软的研究：将权重量化到 1-bit（只有 -1, 0, 1）：

```
传统 INT4: 每个权重 4 bit
BitNet:    每个权重 1.58 bit (三值: -1, 0, 1)

7B 模型: 14GB (FP16) → 1.4GB (1-bit)
```

**状态：** 研究阶段，需要重新训练，不能对现有模型后量化。

### FP4 (Float 4-bit)

比 INT4 更灵活的 4-bit 浮点格式：

```
FP4 格式: 1 bit 符号 + 3 bit 数值
表示范围: 更适合正态分布的权重
```

### 混合精度的精细化

不是"所有层用同一种精度"，而是每个权重矩阵独立选择最优精度：

```
Layer 1 Attention W_q: 4.2 bit (平均)
Layer 1 Attention W_k: 3.8 bit
Layer 1 FFN W_up:      4.5 bit
...
```

## 12.4 推理框架趋势

### 统一推理栈

趋势是向统一的推理框架发展：

```
当前状态:
  llama.cpp → GGUF 格式
  vLLM → SafeTensors 格式
  MLX → MLX 格式
  TensorRT-LLM → 引擎格式

未来趋势:
  统一格式 + 统一 API + 自动硬件适配
```

### 编译器优化

自动将模型编译为最优的硬件特定代码：

```python
# 类似 TVM 的自动优化
import torch
model = torch.compile(model, mode="max-autotune")

# 或使用专门的 LLM 编译器
from llm_compiler import compile
optimized_model = compile(model, target="nvidia-rtx3060")
```

### 推理即服务 (Inference as a Service)

```
本地推理服务:
  Ollama → 一键部署
  LM Studio → 图形界面
  LocalAI → 兼容 OpenAI API

未来:
  模型市场 + 一键部署 + 自动优化
```

## 12.5 投机解码的进化

### 自适应投机

根据任务动态调整投机策略：

```
代码生成: 高接受率 → 每次猜 8 个 token
对话:     中接受率 → 每次猜 5 个 token
数学推理: 低接受率 → 每次猜 2 个 token
```

### 多模型协作

多个不同大小的模型协同工作：

```
小模型 (1B):  快速生成草稿
中模型 (7B):  验证并修正
大模型 (70B): 最终确认

按需调用，最大化效率
```

## 12.7 推理优化的理论前沿

### 计算复杂度的下界

```
问题: 自回归生成是否有更快的算法？

已知: 生成 n 个 token 至少需要 O(n) 次模型调用
探索: 能否在单次调用中生成多个 token？

投机解码: 用小模型预测 + 大模型验证 (工程方法)
Medusa:   多头预测 (架构方法)
Lookahead: Jacobi 迭代 (数学方法)
```

### 信息论视角

```
每个 token 携带的信息量是有限的
理论上，可以通过并行生成来逼近信息论极限

当前瓶颈: 模型的确定性太强 (大部分 token 可预测)
未来方向: 利用可预测性来跳过不必要的计算
```

## 12.10 总结：推理加速的核心思想

回顾全书，推理加速的核心思想可以归结为三点：

### 1. 减少数据搬运

```
权重太大 → 量化压缩
KV Cache 太大 → KV 量化 + GQA + 滑动窗口
重复搬运 → 批处理 + 缓存
```

### 2. 提高计算效率

```
GPU 利用率低 → 批处理 + 算子融合
计算冗余 → 投机解码 + 跳层
架构低效 → 选择高效架构 (GQA, MoE)
```

### 3. 系统级优化

```
硬件未充分利用 → 频率锁定 + 性能模式
内存管理低效 → 大页内存 + 预加载
软件配置不当 → 编译优化 + 参数调优
```

## A.2 llama.cpp 常用命令

```bash
# 编译
make -j                          # CPU only
GGML_CUDA=1 make -j              # + NVIDIA GPU
GGML_METAL=1 make -j             # + Apple Metal
GGML_HIP=1 make -j               # + AMD GPU

# 推理
./llama-cli -m model.gguf \
    -p "prompt" \                 # 输入提示
    -n 256 \                      # 生成 token 数
    -ngl 99 \                     # GPU 层数 (99=全部)
    -t 12 \                       # CPU 线程数
    -c 4096 \                     # 上下文长度
    -ctk q8_0 \                   # KV Cache Key 量化
    -ctv q8_0 \                   # KV Cache Value 量化
    --temp 0.7 \                  # 温度
    --mlock \                     # 锁定内存
    --no-mmap                     # 不使用 mmap

# 量化
./llama-quantize input.gguf output.gguf Q4_K_M

# API 服务
./llama-server -m model.gguf \
    --port 8080 \
    -ngl 99 \
    -np 8 \                       # 并发 slot 数
    --cont-batching               # 连续批处理

# 投机解码
./llama-cli \
    -m large_model.gguf \         # 大模型
    -md small_model.gguf \        # 草稿模型
    --draft-max 5                 # 每次猜 5 个 token
```

## A.4 vLLM 常用配置

```python
from vllm import LLM, SamplingParams

# 基本配置
llm = LLM(
    model="Qwen/Qwen2.5-7B-Instruct",
    tensor_parallel_size=1,           # GPU 数量
    gpu_memory_utilization=0.92,      # 显存利用率
    max_model_len=8192,               # 最大上下文
    max_num_seqs=32,                  # 最大并发
    max_num_batched_tokens=8192,      # 批次最大 token 数
    swap_space=4,                     # CPU swap (GB)
    enable_prefix_caching=True,       # 前缀缓存
    dtype="auto",                     # 数据类型
    trust_remote_code=True,           # 信任远程代码
)

# 采样参数
params = SamplingParams(
    temperature=0.7,
    top_p=0.9,
    max_tokens=512,
    stop=["</s>"],
)

# 推理
outputs = llm.generate(["Hello"], params)
```

建议:

  - 频率越高越好 (CPU 推理时)
```

## C.1 为什么要标准化测试

不同人的"速度"报告差异巨大，原因包括：
- 测量方法不同（首 token vs 稳定生成）
- 测试条件不同（Prompt 长度、生成长度）
- 硬件状态不同（温度、功耗限制）
- 软件配置不同（量化精度、批处理大小）

标准化测试可以让你的结果可复现、可比较。

### 硬件状态

# 4. 使用相同的推理参数
```

## C.3 标准测试流程

### 测试脚本

```bash
#!/bin/bash
# benchmark.sh - 标准化性能测试

MODEL=$1
WARMUP_PROMPT="Hello, this is a warmup."
TEST_PROMPT="Explain the theory of relativity in detail, covering both special and general relativity, their key postulates, and practical applications."
GENERATE_TOKENS=256
WARMUP_RUNS=3
TEST_RUNS=5

echo "=== 性能基准测试 ==="
echo "模型: $MODEL"
echo "测试 Prompt 长度: $(echo -n "$TEST_PROMPT" | wc -c) 字符"
echo "生成 Token 数: $GENERATE_TOKENS"
echo ""

# 预热
echo "预热中 ($WARMUP_RUNS 次)..."
for i in $(seq 1 $WARMUP_RUNS); do
    ./llama-cli -m "$MODEL" \
        -p "$WARMUP_PROMPT" \
        -n 50 \
        -ngl 99 \
        --temp 0 \
        2>/dev/null
done

# 正式测试
echo "正式测试 ($TEST_RUNS 次)..."
TOTAL_SPEED=0
for i in $(seq 1 $TEST_RUNS); do
    OUTPUT=$(./llama-cli -m "$MODEL" \
        -p "$TEST_PROMPT" \
        -n $GENERATE_TOKENS \
        -ngl 99 \
        --temp 0 \
        2>&1)
    
    SPEED=$(echo "$OUTPUT" | grep "eval rate" | grep -oP '[\d.]+(?= tokens per second)')
    echo "  第 $i 次: $SPEED tokens/s"
    TOTAL_SPEED=$(echo "$TOTAL_SPEED + $SPEED" | bc)
done

AVG_SPEED=$(echo "scale=2; $TOTAL_SPEED / $TEST_RUNS" | bc)
echo ""
echo "平均速度: $AVG_SPEED tokens/s"
```

## C.4 指标定义

### 核心指标

```
1. Prompt Evaluation Speed (Prefill 速度)
   = 输入 token 数 / Prefill 耗时
   单位: tokens/s
   
2. Generation Speed (生成速度)
   = 生成 token 数 / 生成耗时
   单位: tokens/s
   
3. Time to First Token (TTFT)
   = 从请求到第一个输出 token 的时间
   单位: ms
   
4. Inter-Token Latency (ITL)
   = 相邻两个 token 之间的时间
   单位: ms
   ITL = 1000 / Generation Speed
   
5. Total Latency (总延迟)
   = 从请求到完整输出的时间
   单位: s
```

### 辅助指标

```
6. GPU Memory Usage (显存占用)
   单位: GB
   
7. CPU Memory Usage (内存占用)
   单位: GB
   
8. GPU Utilization (GPU 利用率)
   单位: %
   
9. Power Consumption (功耗)
   单位: W
```

## C.6 常见测试陷阱

### 陷阱一：没有预热

```
问题: 首次运行比后续慢 20-50%
原因: GPU 频率爬升、缓存冷启动
解决: 先跑 2-3 次预热，再正式测试
```

### 陷阱二：测量不完整

```
问题: 只测了生成速度，没测 Prefill
影响: 长 Prompt 场景的性能被高估
解决: 分别报告 Prefill 和 Generate 速度
```

### 陷阱三：测试条件不一致

```
问题: 不同配置用了不同的 Prompt 或生成长度
影响: 结果不可比较
解决: 使用相同的测试 Prompt 和生成长度
```

### 陷阱四：忽略显存占用

```
问题: 只看速度，不看显存
影响: 可能 OOM 或影响其他应用
解决: 同时报告显存占用
```

### 陷阱五：没有多次测量

```
问题: 只测一次，结果有随机性
影响: 波动大，不可靠
解决: 至少测 3 次，取平均值
```

### 自定义测试脚本

```python
#!/usr/bin/env python3
"""自动化性能测试脚本"""

import subprocess
import json
import time

def run_benchmark(model_path, prompt, tokens, gpu_layers, threads):
    """运行单次测试"""
    cmd = [
        "./llama-cli",
        "-m", model_path,
        "-p", prompt,
        "-n", str(tokens),
        "-ngl", str(gpu_layers),
        "-t", str(threads),
        "--temp", "0",
    ]
    
    result = subprocess.run(cmd, capture_output=True, text=True)
    output = result.stderr
    
    # 解析输出
    metrics = {}
    for line in output.split('\n'):
        if 'prompt eval' in line:
            # 提取 Prefill 速度
            pass
        if 'eval rate' in line:
            # 提取生成速度
            speed = float(line.split()[3])
            metrics['generation_speed'] = speed
    
    return metrics

def main():
    model = "model.gguf"
    prompt = "Explain quantum computing."
    
    configs = [
        {"gpu_layers": 99, "threads": 12},
        {"gpu_layers": 0, "threads": 12},
        {"gpu_layers": 99, "threads": 4},
    ]
    
    for config in configs:
        print(f"\n=== Config: {config} ===")
        for i in range(5):
            metrics = run_benchmark(
                model, prompt, 256,
                config["gpu_layers"],
                config["threads"]
            )
            print(f"  Run {i+1}: {metrics.get('generation_speed', 'N/A')} tok/s")

if __name__ == "__main__":
    main()
